# 多写者共享聚合文件须读-合并-写，禁止从零整体重写

> 同一份聚合/快照/注册表文件被多个独立写入者共写时，任一写入者若「按自己负责的数据从零拼出全文再覆盖写」，会无声抹掉别人写的条目——文件不报错、内容也合法，但对端数据丢失，往往要等对端流程下次运行才暴露。

## 现象

一个文件被多个独立流程写入（不同模块 / 平台 / 步骤 / 进程）。其中某个流程用「整体重写」方式更新：先按自己这一摊数据在内存里拼出完整文件内容，再覆盖写盘。结果是这次没参与拼接的其他流程的条目被一并清掉。表象具有迷惑性——写操作成功、文件语法合法、本流程的数据也对，只有对端那部分凭空消失。多写者顺序执行时还会互相覆盖，最终只剩最后一个写入者那一份。

## 适用条件

- 目标文件有 ≥2 个独立写入路径（如双平台、CI 与本地、多个 skill / 脚本各写一段）。
- 至少一个写入路径采用「从零拼全文 + 覆盖写」而非「就地增量更新」。
- 典型文件：安装/资产快照、lock 文件、注册表、索引、合并配置。

## 规避 / 操作

| 场景 | 做法 |
|---|---|
| 文件只有单一写入者 | 整体重写无妨 |
| 文件被多写者共写 | **禁止从零整体重写**；改「读-合并-写」：写前先读回现有内容，保留不属于本次职责的条目，再连同本次重算的条目一并写 |
| 区分「哪些条目不归我管」 | 用稳定标志判归属（如条目内的平台/来源字段），抽成共用「拆分现有条目」函数让各写入者复用，避免各写一份解析逻辑漂移 |
| 写完之后 | 断言对端条目仍在；补 RED→GREEN 测试——RED 先复现「整体重写丢对端」，GREEN 证明合并后两端都在 |
| 注释/文档声称「会保留 / 会合并」 | 不等于代码真做了（doc/code 漂移），以可执行测试为准，发现漂移按 bug 处理 |

## 原理

「整体重写」隐含一个假设：写入者知道整份文件「应该长什么样」。但多写者场景下没有任何单一写入者掌握全文——每个只知道自己负责的那部分。把「全文真相」留在文件本身、每次写入只做「读回 → 合并自己这部分 → 写回」，才能让各写入者互不踩踏。这本质上是并发/分治写入下的幂等合并问题，与「整份覆盖」是两种相反的心智模型。

## 一句话总结

> 多写者共享的聚合/快照文件，任何一次更新都要「读-合并-写」而非「从零整体重写」：写前读回保留非本职责条目，并用测试断言对端数据没被抹掉。
